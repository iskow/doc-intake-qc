## Working title
Document Intake QC Agent

## The problem
SMEs and legal/ops teams receive messy file dumps — bad names, duplicates, wrong formats — and QC them by hand or not at all.

## The build (updated as phases close)
- _Planned approach:_ Python automation that takes a folder of messy mock client files, validates and classifies them, flags exceptions, and outputs an eDiscovery-grade QC report.
- **Phase 0 (done):** Built a realistic 41-file "client dump" test fixture — invoices, contracts, correspondence, reports, photos — with 21 deliberately planted problems across 9 categories (duplicates, misleading extensions, empty files, impossible dates, junk files, naming chaos). The fixture and its answer key are generated from the same script, so the test can't lie: every planted error is known, and the pipeline's catch rate can be measured exactly.
- **Phase 1 (done):** Built the inventory scanner — a read-only walk of the folder that produces a `manifest.csv`: one row per file with its true type (read from the file's actual bytes, not its name), size, SHA-256 fingerprint, and dates. This is the evidentiary base every later check builds on. It already exposes the traps a name-based sort would miss: a PNG wearing a `.jpg` name, plain text renamed `.pdf`, empty files, and copy-paste duplicates that share a fingerprint.

- **Phase 2 (done):** Built the validation rules engine — nine deterministic checks that turn the inventory into an exceptions report. It finds byte-identical duplicates, files whose extension lies about their content, empty files, near-duplicate names where two versions disagree (which one is authoritative?), impossible dates, OS junk, unexpanded archives, naming-convention breaks, and files buried too deep to survive a manual review. It also reads each document's **embedded author** — distinct from the party named in the filename and from the custodian who supplied it, a distinction most tools blur. Every finding carries a severity and a plain-English reason.

- **Phase 3 (done):** Added the AI classification layer — every readable document is sorted into invoice, contract, correspondence, report, or other by a language model that runs **entirely on the local machine**. No document is ever uploaded to a third-party service, and the running cost is $0. Two design choices carry the weight: the model is shown the document's **text only**, never its filename or folder — so the folder stays an independent second opinion, and the tool can flag documents filed in the wrong place. And it never guesses: a photo or an empty file has no correct label, so it gets none and goes to a review queue instead.

- **Phase 4, part 1 (done):** Built the non-destructive organizer — it copies the whole messy dump into a clean folder structure, sorted by whichever axis the client needs: **what the document is** (the AI label), **who wrote it** (the embedded author), or **where it was collected from** (the custodian). Originals are never renamed, moved, or touched; the tool refuses outright to write anything inside the source folder. Files with no answer on the chosen axis go to an `Unassigned` folder rather than being guessed into a bucket that looks tidy and is wrong.

- **Phase 4, part 2 (done):** Made the copy **metadata-faithful**. A reorganized set is worthless in a legal matter if the copy quietly stamps every file with today's date — date filtering is one of the first things done to a collection, and a wrong date doesn't fail loudly, it produces plausible wrong results. Every copy now keeps its original creation and modified dates, its file permissions, and its place in the client's original folder structure, and every run writes a cross-reference table recording exactly where each document came from.

- **Phase 4, part 3 (done):** Built the deliverable the client actually reads — a single **self-contained HTML report** that opens in any browser with no internet connection, no install, and nothing to configure. It leads with the number that matters (25 of 41 files need attention), puts every finding into **one review queue** in severity order rather than splitting "rules" and "AI" into two lists nobody cross-references, explains in plain language what each rule catches and why it matters, and closes with a limitations section stating what the tool did *not* establish.

- **Phase 5, part 1 (done):** Made it runnable by someone who is not me. One command runs the whole pipeline on the mock intake in about 45 seconds and opens a finished report. The step order is not written in instructions a reader has to follow correctly — it lives in the code, and the build fails if the documentation and the code ever disagree about it.

## Results & proof
- **Phase 5 QC gate: 85 automated checks, 0 failures** (`scripts/qc_phase5.py`). The whole pipeline runs from a cleared working tree — every generated file deleted first, so "it works" means "it works from nothing" rather than "it works on top of yesterday's output" — and the client's originals are fingerprinted before and after and come back unchanged.
- **The README cannot go stale, because the build checks it.** Every command it documents is compared against the code that actually runs, in order; every file it links to must exist; every flag it shows must be one the script accepts; and the ten headline figures in its results table are recomputed from the live pipeline on every run. Changing one number from 25 to 26 fails the build, naming the figure. Documentation that drifts from the tool is the normal outcome, and it is invisible — a README renders perfectly no matter how wrong it is.
- **The screenshot cannot go stale either.** The report itself is regenerated per run and deliberately not committed, so the committed images are the only way the repository can show its own deliverable. Each capture records the figures the report was showing at the time, and the build compares them against what the pipeline produces now — so if the results ever change, the build fails rather than leaving an image quietly telling every reader an old number. Proven by editing one figure: exactly one failure, naming it and printing the command that fixes it.
- **A documentation claim was disproved by its own test and corrected.** The README stated that generating the report too early "does not fail loudly." Deliberately reordering the pipeline to check that produced 21 failures — and showed the report refusing to render at all, naming the missing input. The sentence had been written from reasoning rather than measurement; the test caught it, and the README now says what actually happens.
- **Two of the new checks were found to be broken by running them.** One flagged its own source file as unfinished work, because the words it searches for appeared in its own code. The other reported a folder as "not excluded from the repository" whenever the folder was absent — which is precisely the state the check runs in, so it would have failed for a reason that had nothing to do with the thing it was testing. Both were diagnosed, fixed, and the diagnosis verified directly rather than assumed.
- **Phase 4 QC gate: 131 automated checks, 0 failures** (`scripts/qc_phase4.py`) — up from 73, with 58 covering the report alone.
- **The report proves its own arithmetic, in front of the client.** Eight reconciliation checks run before the file is written, and a mismatch stops it rendering rather than printing a caveat: flagged plus clean must equal files received, severities must account for every finding, each organized copy must contain every file. The results are printed *into* the report as a table, so the client sees the check rather than being asked to trust the total.
- **The QC gate never asks the report whether it is right.** It recomputes 21 figures from the source data and compares each against the number read back out of the finished HTML. Two sides, two routes, same answer — otherwise a single bug tells itself it is correct and both agree. Proven by deliberately inflating one headline number: the gate caught it (`HTML says 26, recomputed 25`) *while the report's own internal checks still passed*, which is exactly the blind spot the design exists to cover.
- **Honesty is enforced by the build, not by good intentions.** The report's limitation statements — access times can't be preserved, ownership is unproven, bad filenames were reported but never fixed — are asserted to be **present**. For these the failure mode isn't saying something false, it's saying nothing: an edit that deletes an inconvenient caveat leaves a cleaner-reading report that over-claims. Proven by swapping the access-time caveat for the plausible, false "All timestamps are carried through faithfully" — the build failed.
- **The date-fidelity claim re-proves itself on every run.** Rather than printing a result measured once, the report re-checks every copy against the dates recorded before any copy existed — **246 comparisons, all matching**. If the set is ever re-copied with a tool that loses dates, the next report says so instead of repeating an old promise.
- **The report's colors were computed, and the brand's own palette failed.** Run through a contrast and colorblind-safety validator instead of being eyeballed: the intended soft brand color measured **1.69:1** against the page — effectively invisible — and the intended accent measured **2.44:1**, below the minimum for a chart bar. Both were corrected before anyone saw the report. Severity is shown as one color getting darker rather than red/amber/green, and the words "high / medium / low" are always printed — so it survives a colorblind reader, a grayscale printout, and a bad projector.
- **Phase 4 parts 1–2: 73 checks, 0 failures.** All 41 files were copied into all three structures — **123 copies, every one byte-identical to its original**, with **creation dates, modified dates, folder structure and file permissions preserved**, so the impossible-date findings survive the copy instead of being silently reset to today. The originals were fingerprinted before and after and are unchanged. Nothing was dropped, duplicated, or overwritten.
- **The copy method was chosen by measurement, not by reputation.** Four copy methods were tested against a file carrying a known creation date, a non-inherited permission entry and a hidden data stream, and the results decided the design: the standard Python copy — the obvious choice, and the one the tool originally used — silently loses creation dates and replaces file permissions with whatever the destination folder hands out. It looks like it worked. Nothing warns you.
- **One limitation is reported rather than quietly failed.** This phase was planned to preserve all three file timestamps. Measuring showed the third one — last-accessed — *cannot* be preserved: reading a file updates it, so the act of fingerprinting the collection destroys it before any copy happens. The requirement was withdrawn and written up as a finding, with the practical consequence for clients: access dates must be captured by the collection tool at acquisition, or they are gone. A tool that had claimed all three would have been believed.
- **One document, three different right answers.** A service agreement named for Acme Supply was *written* by someone at Meridian and *collected* from Legal. Sorted by content it files under contracts, by author under Dana Cruz, by custodian under Legal — and the tool keeps all three straight instead of collapsing them into one "Acme" folder. Most filing tools blur exactly this distinction; in a legal or compliance context, blurring it is how the wrong document gets produced to the wrong party. The QC gate asserts the three answers stay different, so the claim can't quietly stop being true.
- **The tool refuses to guess, and refuses to touch originals.** Nine files have no content label and land in `Unassigned` rather than a plausible-looking bucket. The organizer also blocks any attempt to write into the source folder — proven not just by checking that it blocks, but by confirming it still *allows* legitimate destinations, including catching a disguised path that pointed back into the source folder through a `..` shortcut. A safety check that blocks everything is indistinguishable from one that works.
- **The gate was broken on purpose, five times.** It passed on its first run — which is when a test deserves the least trust. Each claim was disproved in isolation: switching the copy method so creation dates were lost failed all three modes (while modified dates still passed, proving the right check was doing the work); flattening the folder structure produced 15 failures; dropping the permission-copying flag produced **exactly one**; making the tool guess a label for unreadable files failed it again, naming all nine. Every break was reverted and re-verified.
- **One test exposed a check that was passing for the wrong reason, and it was downgraded rather than kept.** Removing the flag that copies file *ownership* didn't fail the gate — because every file on the test machine has the same owner, so the check could never have caught a real problem. It now reports the narrower thing it actually proves, and the limitation is written down. Finding out your own test is hollow is only useful if you act on it.
- **Phase 3 QC gate: 17 automated checks, 0 failures** (`scripts/qc_phase3.py`). **Every one of the 32 readable documents was classified correctly** — including two deliberate traps: a contract hiding under the scanner-default name `scan_001.pdf`, and a report sitting loose at the root as plain text wearing a `.pdf` extension. Both were caught by content, which a name-based or folder-based sort would have filed wrong. The other 9 files have no readable text and were all reported unclassified — **not one invented label**. Runtime: 41 files in about 43 seconds on a consumer desktop GPU, at $0.
- **The model was measured before the pipeline was built, not after.** A throwaway spike classified 12 documents first; only after it scored 12/12 was any production code written. That spike also caught a bug worth naming: an early text extractor was scraping 541 characters of binary noise out of a solid-color PNG and feeding it to the model, which then confidently labeled it. Building first and testing later would have shipped that.
- **The Phase 3 gate was tested three ways.** Two of the new rules find nothing on this data set — the model agrees with the folder structure everywhere and is never unsure — and a rule that never fires is indistinguishable from a rule that is broken. So both were proven by planting a deliberate error and confirming each one fired. Then the extraction bug was reintroduced on purpose: the gate failed with a non-zero exit and named all five photos and the archive as files that had been given invented labels.
- **Phase 2 QC gate: 22 automated checks, 0 failures** (`scripts/qc_phase2.py`). On the 41-file dump: **30 exceptions across 23 files, 18 files clean.** All 21 planted errors were caught by the rule responsible for them — and, just as important, **not one clean file was falsely flagged.** Author extraction matched the answer key on all 41 files. Totals reconcile in code against both the folder and the answer key.
- **The gate itself was tested.** A rule was deliberately switched off to confirm the QC gate *fails* when something is missed — it did, naming the exact two files that slipped through. A passing test only means something if it can fail.
- Phase 0 QC gate: 39 automated checks, 0 failures (`scripts/qc_phase0.py`). The *test harness itself* is eDiscovery-grade — you can't claim a QC tool works unless you know exactly what it should catch.
- Phase 1 QC gate: 13 automated checks, 0 failures (`scripts/qc_phase1.py`). Row count reconciles with the folder and the answer key; every file is accounted for (nothing missed, nothing invented); duplicate pairs share a hash; and the scan is proven read-only by fingerprinting every file before and after — the originals are never touched.

## What this demonstrates
- **Data rigor first, AI second.** The tool identifies files by their content bytes and a cryptographic hash, not by trusting the filename — the same defensible, evidence-first approach used in eDiscovery. Duplicates are caught by fingerprint, not guesswork.
- **Chain-of-custody mindset.** The scanner is provably non-destructive (verified in code), so a client's originals are safe — the credibility a legal/ops or bookkeeping client needs before handing over a data set.
- **Precision matters as much as recall.** Catching every problem is easy if you flag everything. This engine caught 21 of 21 planted errors while leaving all 18 clean files untouched — so the exceptions list is a work queue, not noise a client has to re-sort by hand.
- **Findings are explainable, not black-box.** Every flag names the rule, a severity, and a concrete reason ("extension '.doc' but content is 'text'"). Deterministic rules come first precisely because they're defensible; the AI layer is added on top, never underneath.
- **Confidential documents never leave the client's machine.** The classifier runs a local open-source model, so there is no API key, no vendor account, and no upload — the question a legal, compliance, or bookkeeping client asks before anything else.
- **The AI is not trusted on its own word.** The model reports high confidence on nearly every document, including junk — so confidence alone was measured, found unusable as a safety check, and backed up with something independent: the label read from the content is compared against the folder the file was filed in, and disagreement becomes a finding. Measuring that instead of assuming it is the difference between a demo and a tool.
- **Three questions, kept separate.** "Who wrote this," "who gave it to us," and "what is it" are three different questions that most tools answer as one. Author is metadata inside the file; custodian is where it was collected from and comes from the client's own collection log, never guessed from a filename; content type is read from the text. The tool sorts by any of the three and never lets one stand in for another.
- **The gaps are stated, not hidden.** The organizer deliberately does *not* rename files: renaming is the one step that can't be undone by looking, and keeping original names keeps every copy traceable to what the client actually sent. The consequence — bad filenames are reported but not fixed — is written into the report rather than left for the client to discover. Knowing what a tool doesn't do is part of knowing what it does.
- **It says "I don't know" when it doesn't know.** Nine files have no readable text, and the tool refuses to label them rather than producing a plausible guess. In a review set, a confident wrong answer costs more than an honest gap.
